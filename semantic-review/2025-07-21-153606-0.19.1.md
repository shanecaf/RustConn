# SSH password polling and RDP certificate mismatch handling

RustConn 0.19.1 addresses two user-facing reliability issues. The SSH auto-fill fix replaces a single 120ms one-shot timer with a 150ms repeating poll (10s deadline) to handle VTE's failure to emit further signals after a no-echo prompt lands. The RDP change introduces a `FreerdpFailure` enum that distinguishes certificate mismatch from generic errors, routes it through a new `CertChangedCallback`, and shows a confirmation dialog that clears FreeRDP's TOFU store before reconnecting.

Watch for: the SSH polling timer has no cancellation path tied to tab/session lifetime â€” if the user closes the tab during the 10s window, the timer continues firing against a destroyed session (confirmed). The RDP certificate pattern match has an operator-precedence subtlety that works correctly but is fragile (confirmed). The `known_hosts2` removal uses substring matching that could over-match hostnames sharing a prefix (likely).

## High-level view

The SSH polling fix is structurally sound: `inject_once` already guards against double-injection via `password_sent`, and the `recheck_scheduled` flag prevents timer pile-up from repeated signal firings. The remaining gap is session lifetime â€” glib timers outlive the closure's captured `Rc` references to the notebook/session, and VTE widget destruction does not automatically cancel them.

The RDP certificate flow is well-integrated into the existing callback architecture. `classify_freerdp_failure` now returns a typed enum, the watchdog dispatches to either the error callback or the new cert-changed callback, and the UI layer shows an `adw::AlertDialog` via `show_confirm`. The cert store cleanup covers both FreeRDP 2.x and 3.x paths. The one design concern is a substring-match in `remove_from_known_hosts2` that relies on the assumption that host strings are never substrings of other hosts in the file.

1. **Timer outlives session** â€” The 150ms polling timer and 10s deadline timer are not cancelled when the SSH tab is closed. If the session/notebook widget is destroyed mid-poll, `inject_once` calls `detect_password_prompt` / `send_text_to_session` on a dead session. Store the `glib::SourceId` and disconnect it in the tab-close path, or use `@weak` references that make the closure no-op on widget destruction.
2. **Comment says "exponential backoff" but interval is fixed** â€” The polling comment at both sites says "retries with exponential backoff" but the actual implementation is a fixed 150ms interval. Update the comment to match reality.
3. **Operator precedence in cert pattern match** â€” `joined.contains("Certificate") && joined.contains("denied")` binds `&&` tighter than the surrounding `||`, which is the intended behavior, but clippy's `precedence` lint may flag it in future versions and it's easy to mis-read during maintenance. Explicit parentheses cost nothing and prevent accidents.
4. **Substring match in `remove_from_known_hosts2`** â€” `line.contains("{host} {port}")` matches any line containing that substring, not just lines starting with it. A host `10.0.0.1` would match a hypothetical line containing `110.0.0.1 3389` embedded in a longer field. Use `line.starts_with(&host_pattern)` or split on whitespace and compare the first two fields to the known format `<host> <port> <fingerprint> ...`.

## SSH polling timer lifecycle

The polling timer captures `inject_once` (which holds `Rc<SharedNotebook>` and `session_id`) and `password_sent_poll` (a `Cell<bool>`). When the tab is closed, the notebook removes the session, but the glib source remains registered on the main loop. On the next 150ms tick, `inject_once` calls `detect_password_prompt(&notebook_clone, session_id)` â€” if the notebook's internal map no longer has that session, this is likely a no-op (returns `None`/`false`), but it depends on the notebook implementation not panicking on a stale UUID lookup.

The 10s deadline timer has the same issue: `password_sent_deadline.set(true)` is harmless on its own, but if later code checks `password_sent` for other cleanup decisions, the forced `true` from a deadline on a closed tab could confuse post-mortem diagnostics.

In both the `start_ssh_connection_internal` and `reconnect_ssh_in_place` paths, the code is duplicated verbatim. If a fix is applied to the timer lifecycle, it needs to be applied in both places â€” this is a maintenance hazard that a shared helper would eliminate.

## RDP certificate detection patterns

The `classify_freerdp_failure` match clause:

```rust
} else if joined.contains("certificate not trusted")
    || joined.contains("does not match the certificate used for previous connections")
    || joined.contains("Certificate") && joined.contains("denied")
{
    FreerdpFailure::CertificateMismatch(...)
}
```

The third arm (`Certificate && denied`) catches the legacy FreeRDP 2.x message. Operator precedence is correct (`&&` binds tighter, so it's `A || B || (C && D)`), but explicit parentheses would make the intent obvious and protect against accidental refactoring.

This clause was previously routing to `FreerdpFailure::Error` with the message "Connection rejected: server certificate was not accepted." By reclassifying it as `CertificateMismatch`, the change assumes that *all* certificate denials are recoverable by removing the stored cert. If a certificate is denied for a different reason (e.g., expired CA, self-signed cert rejected by policy), the dialog will still offer "Accept new certificate" â€” which may not resolve the underlying problem. This is an acceptable UX trade-off for 0.19.1 since `/cert:tofu` covers the common case, but a future improvement could distinguish "changed" from "untrusted" by inspecting the specific FreeRDP error code.

## `known_hosts2` substring matching

The FreeRDP `known_hosts2` format is: `<host> <port> <fingerprint> <subject> <issuer>`, one entry per line (verified against the actual file on this system). The removal logic uses:

```rust
let host_pattern = format!("{host} {port}");
let filtered: Vec<&str> = content
    .lines()
    .filter(|line| !line.contains(&host_pattern))
    .collect();
```

Since the host is always the first field and the port the second, `line.starts_with(&format!("{host} {port} "))` (note trailing space) would be more precise. The current `contains` approach works for typical IP/hostname entries but could false-positive on base64-encoded subject/issuer fields that happen to embed the pattern. The risk is low in practice but the fix is trivial.

## Callback lifecycle in the RDP watchdog

The `arm_external_exit_watchdog` function now takes `on_cert_changed: Rc<RefCell<Option<CertChangedCallback>>>` alongside the existing `on_error` and `on_state_changed`. The take-invoke-restore pattern is correctly applied for the cert callback (matching the existing pattern for error callbacks), preventing `BorrowMutError` panics if the callback re-enters the widget.

The host and port are passed as owned `String` and `u16` â€” extracted from `fallback_config` with `map_or_else(String::new, |c| c.host.clone())` and `map_or(3389, |c| c.port)`. An empty host string would pass through to `remove_known_certificate`, which would construct a pattern like ` 3389` and potentially remove wrong entries. This path is only reachable if `fallback_config` is `None` at the time the watchdog fires, which would indicate a logic error elsewhere â€” the cert dialog would show a blank host in the message, making it obvious something is wrong.
